# 修复文件内容边界问题计划

## 问题描述

用户报告了以下文件内容写入问题：

1. **volume_outline 中的黄金三章内容被写入了 book_rules.md**
2. **book_rules 的部分内容被写入了 current_state.md**
3. **current_state 的部分内容被写入了 pending_hooks.md**
4. **story_bible.md 中没有过滤 LLM `<think>` 标签的思考内容**
5. **story_bible.md 中有一部分内容被截断，导致不完整**

## 根本原因分析

### 问题 1-3: 内容边界混淆
- 在 `architect.ts` 中，LLM 生成多个文件内容时，可能使用了相似的标记或格式
- 解析逻辑无法准确区分不同部分的开始和结束
- 正则表达式或字符串分割逻辑不够精确

### 问题 4: `<think>` 标签过滤不完整
- 虽然代码中有过滤逻辑（第 824-830 行），但可能：
  - 标签格式不一致（如 `<think>`, `<thinking>`, 或属性形式 `<think ...>`）
  - 嵌套标签处理不当
  - 某些生成路径没有应用过滤

### 问题 5: 内容截断
- `maxTokens: 8192` 限制可能导致长内容被截断
- 没有检查内容完整性
- 缺少分段生成或续写机制

## 解决方案

### 阶段 1: 增强内容解析和边界检测

#### 1.1 改进文件内容解析器
**文件**: `packages/core/src/agents/architect.ts`

- 使用更明确的 XML 标签格式（如 `<file name="xxx">...</file>`）
- 添加内容校验和完整性检查
- 实现模糊匹配和纠错机制

#### 1.2 添加内容验证层
**文件**: `packages/core/src/utils/content-validator.ts` (新建)

- 验证每个文件内容的格式正确性
- 检测内容重叠和边界问题
- 提供自动修复建议

### 阶段 2: 完善 `<think>` 标签过滤

#### 2.1 统一过滤逻辑
**文件**: `packages/core/src/utils/llm-output-filter.ts` (新建)

- 创建统一的过滤函数，处理所有 LLM 输出
- 支持多种标签格式：`<think>`, `<thinking>`, `<thought>`, 以及带属性的变体
- 处理嵌套标签

#### 2.2 应用过滤到所有输出点
**文件**: `packages/core/src/agents/architect.ts`, `packages/core/src/agents/writer.ts` 等

- 确保所有 LLM 输出都经过过滤
- 在写入文件前统一处理

### 阶段 3: 解决内容截断问题

#### 3.1 增加内容完整性检查
**文件**: `packages/core/src/agents/architect.ts`

- 检查生成的内容是否完整（如 XML 标签是否闭合）
- 检测明显的截断特征（如未闭合的括号、引号等）

#### 3.2 实现续写机制
**文件**: `packages/core/src/agents/architect.ts`

- 当检测到内容截断时，自动请求续写
- 合并多次生成的内容

### 阶段 4: 添加调试和监控

#### 4.1 增强日志记录
- 记录每个文件内容的生成长度
- 记录解析过程中的边界检测结果
- 记录过滤前后的内容差异

#### 4.2 添加内容哈希校验
- 生成内容摘要，便于追踪问题
- 检测重复或异常内容

## 实施步骤

### 步骤 1: 创建内容验证工具
- 创建 `packages/core/src/utils/content-validator.ts`
- 实现内容边界检测算法
- 实现 `<think>` 标签过滤函数

### 步骤 2: 修改 architect.ts
- 在 `generateFoundation` 方法中应用内容验证
- 修复 `storyBible` 的 `<think>` 标签过滤
- 改进多文件内容解析逻辑

### 步骤 3: 修改 runner.ts
- 在 `reviseDraft` 方法中应用内容验证
- 确保修订后的内容正确写入对应文件

### 步骤 4: 添加测试
- 创建单元测试验证内容边界检测
- 测试 `<think>` 标签过滤的各种情况
- 测试内容截断检测

### 步骤 5: 集成验证
- 端到端测试书籍创建流程
- 验证所有文件内容正确性

## 预期结果

1. **文件内容边界清晰**: 每个文件只包含对应类型的内容
2. **无 `<think>` 标签残留**: 所有 LLM 思考内容被正确过滤
3. **内容完整**: 无截断或缺失
4. **可调试**: 通过日志可以追踪内容生成和解析过程

## 风险评估

- **低风险**: 修改主要集中在新工具函数和验证层
- **中风险**: 修改 `architect.ts` 的解析逻辑可能影响现有功能
- **缓解措施**: 添加全面的单元测试和集成测试
